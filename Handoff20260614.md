## 問題二：多台 Mac mini 共用同步資料夾（尚未實作）

**情境**：三台以上 Mac mini 共用同一個同步資料夾（iCloud / Dropbox），各台機器的資料夾路徑可能有些不同，目前會產生路徑錯誤。

**已決定採用方向一（config.local.json 放在 ~/Library/）**：

- 同步資料夾內：`config.json`（共用預設值，進 git）
- 每台機器各自：`~/Library/Application Support/WhisperVoice/config.local.json`（機器專屬覆蓋，不同步）
- App 啟動時先讀 `config.json`，再讀 Library 的 `config.local.json` 覆蓋差異部分

**典型用途**：每台機器的 `input_device`（麥克風名稱）不同，用 `config.local.json` 各自指定。

### 待實作項目（下一個 session 執行）

**1. `_voice_config.py`**：修改 `load_config()` 加入 `config.local.json` 覆蓋邏輯

```
讀取順序：
  config.json（同步資料夾，共用）
  → ~/Library/Application Support/WhisperVoice/config.local.json（本機覆蓋）
  → deep merge，local 的值優先
```

**2. `.venv` 排除同步**：避免 venv 跨機同步造成 binary 衝突
- iCloud 方案：把 `.venv` 改名為 `.venv.nosync`，或用 `xattr -w com.apple.fileprovider.ignore#P 1 .venv`
- 需先確認用者的同步方式（iCloud / Dropbox / 其他）→ **尚未確認**

**3. `install.sh` 更新**：
- 移除舊的「建立 `啟動語音輸入.command`」步驟（已過期，路徑寫死）
- 新增步驟：偵測可用麥克風 → 引導選擇 → 寫入 `~/Library/.../config.local.json` 模板

**4. 新機器首次安裝流程（文件化）**：

## 問題三：Python vs Xcode App 評估（討論完成，結論待定）

### 實測數據（PID 78299，本機實際量測）
- 閒置記憶體：**149 MB**
- venv 磁碟：**23 MB**
- 閒置 CPU：**≈0%**（pynput 在 macOS 是 Quartz event tap，事件驅動）
- 辨識速度：**網路綁定**（STT 1946ms + LLM 860ms），語言無關

### 核心結論
- 資源差異（記憶體 125MB）對 Mac mini 不痛不癢，**不該是決策主因**
- Xcode 真正的價值：多機部署乾淨、安裝簡單（拖 .app）、不怕 macOS 升版弄壞依賴
- Xcode 真正的代價：改邏輯要重新編譯、重寫需 2–4 週

### 決策關鍵（使用者尚未最終決定）
使用者確認：功能穩定，但「**常調詞彙 JSON**」（user_vocab.json、layer1/2）。
詞彙 JSON 不論 Python 或 Swift 都可以熱重載，這點不影響決策。

**尚未釐清**：除詞彙外，是否還會常動 prompt / provider / 修正邏輯？
- 如果會 → 留 Python + 方向一（最快、CP 值最高）
- 如果不會 → 可考慮 Xcode（多機部署一勞永逸）
